# GradeInsight（班级成绩可视化系统）计划表与架构草案

日期：2026-02-25  
基于样例：`2025-2026学年度第一学期期末考试【八年级】(英语)-八年级7班.xlsx`（含 `得分明细` 题目列与复合题子列）

配套 PRD：`docs/requirements/2026-02-25-gradeinsight-prd.md`（以“数据分析平台”为定位，P0/P1/P2 已拆分）

## A. 技术选型（2-3 个方案 + 推荐）

### 方案 A（推荐）：Django + HTMX + Tailwind + ECharts + PostgreSQL
- 优点：一个代码仓库即可完成后台/前台；部署简单；CRUD（考试/学生/任务）开发效率高；适合 iPad 触控与表格场景
- 缺点：前端交互复杂度高时不如 SPA 自由（但本项目用 HTMX/少量 JS 足够）

### 方案 B：FastAPI（后端）+ React/Vue（前端）+ ECharts + PostgreSQL
- 优点：前后端分离、交互更灵活；前端体验更像“应用”
- 缺点：开发/部署链路更长；需要构建前端产物与 API 版本管理

### 方案 C：Next.js 全栈 + PostgreSQL（Prisma）+ ECharts
- 优点：全栈一体、生态成熟、开发体验好
- 缺点：Excel 解析依赖 Node 生态；VPS 运维对你现有环境可能更重

> 先做 MVP 时建议选 A；若后续要多角色/更复杂交互，再升级到 B/C 也可（数据模型可复用）。

## C. MVP 页面清单（兼容 PC + iPad）

1) 登录页  
2) 考试列表页：新建/导入/查看  
3) 考试详情页：总览（均分、分布、排名表）+ 题目筛选入口  
4) 题目筛选页（核心）：题号选择 + 条件/得分输入 → 名单列表  
5) 任务清单页（核心）：来自筛选结果；点名切换“已交/未交”；可筛“仅未交”  
6) 学生详情页（MVP 必须）：总分趋势 + 排名趋势 + 各次考试明细

## D. 数据库设计（建议最小可用模型）

### 核心实体
- `classes`：班级（如“八年级7班”）
- `students`：学生（姓名 + 外部编号：准考证号/自定义考号）
- `exams`：考试（名称/日期/学科/班级/来源文件信息）
- `exam_scores`：每个学生在一次考试的汇总分（总分/客观/主观/班次等）
- `exam_question_scores`：每题（含子问）得分明细（question_key=`51`/`51(3)`，score_value=2.0）
- `tasks`：任务（由“题号+得分筛选”生成，绑定 exam_id + question_key + 条件）
- `task_submissions`：任务提交状态（student_id + task_id + submitted_at）

### 关键约束
- 学生唯一性：`(class_id, external_student_id)` 唯一（`external_student_id` 默认使用 `准考证号`；`自定义考号` 作为辅助字段入库用于校验/追溯）
- 防重复导入：`exams.source_file_hash` 唯一（或 `(class_id, name, date)` 唯一）

## E. Excel 导入与解析策略（结合样例）

1) 上传后先解析 `得分明细` 表头（第 2 行）  
2) 识别基础列：姓名、班级、总分、客观/主观、班次等  
3) 将题目列统一归一化为 `question_key`（字符串）：`1`…`56`、`31(1)`、`51(5)`  
4) 生成“预览结果”：学生数、题目列数、缺失值统计、异常值（非数字），并提示“本次考试不参与统计”的记录（如总分为文本：未扫/缺考等）  
5) 用户确认后入库（事务写入，失败回滚）

### 考试日期口径（已确认）
- `exam_date` 选填；为空时趋势/对比按 `imported_at` 排序，并在 UI 提示“未填写日期”。  

## F. 统计计算与性能策略（建议）

- **写入时计算**：每次导入后预计算基础统计（均分/中位数/分段人数等）并缓存，考试详情打开更快；同时预计算“每题平均分/中位数/样本数N”，用于学生弱势/良好题目判定。  
- **查询时计算**：题目筛选、任务清单等以查询为主（加索引：`exam_id + question_key + score_value`）。  
- **数据质量策略**：对缺失值/非数字值做标记；总分为非数值（如“未扫，不计排名”）的记录**不参与统计**（但保留原始数据可追溯）；页面提示“统计口径/不参与统计人数”。  

## G. 部署计划（公网 VPS）

- 推荐：`docker compose` 运行 `web`（Django/FastAPI/Next.js）+ `db`（PostgreSQL）+ `caddy`（HTTPS 反代）
- 备份：每天定时 `pg_dump` + 上传到对象存储/异地；保留策略（7/30 天）
- 安全：强密码/可选 2FA；限制上传大小；仅登录用户可访问
