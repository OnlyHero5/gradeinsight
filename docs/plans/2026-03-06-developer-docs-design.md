# 开发维护文档补全设计

**日期**: 2026-03-06
**状态**: 已批准，待执行
**范围**: `README.md`、`docs/plans/`、`docs/runbook/`

---

## 背景

`GradeInsight` 当前已经具备较完整的产品需求文档、PRD、README 概览，以及基础部署/备份手册，但从开发维护视角仍存在几处明显空白：

- 新接手开发者缺少一份“系统脑图”，很难快速理解模块边界与主流程。
- 领域模型已经较稳定，但缺少可直接查阅的数据模型与统计口径说明。
- README 侧重功能与快速启动，对“应该改哪里、如何回归、哪些测试覆盖了哪些路径”说明不足。
- 运维文档覆盖了最小部署与备份流程，但环境变量说明、生产检查项与排障入口较少。

---

## 目标

在不重复需求文档与 PRD 的前提下，补出一组面向开发维护者的中文文档，使后续开发者能够：

1. 快速理解 `gradebook` 与 `worklists` 的职责边界。
2. 看懂“导入、分析、筛选、任务追踪”四条主流程如何在代码中落地。
3. 明确核心模型、关键字段、统计口径与去重/状态规则。
4. 按文档完成本地启动、调试、测试和基础运维检查。

---

## 方案对比

### 方案 A：新增 3 份开发维护向文档 + 小幅补强运维文档（采用）

内容：
- 新增系统架构总览
- 新增数据模型与数据流说明
- 新增开发者上手手册
- 补强现有运维文档与 README 的文档索引

优点：
- 与现有 README、需求文档互补，不重复铺陈业务背景
- 直接解决“接手难、定位慢、回归靠猜”的问题
- 文档粒度适中，后续维护成本可控

缺点：
- 会新增多份文档，需要保持同步更新

### 方案 B：只扩写 README

优点：
- 入口集中，查找简单

缺点：
- README 会过长，混合产品说明、开发说明与运维说明
- 难以承载数据模型、数据流、测试地图等细节

### 方案 C：一次性拆成完整文档体系

优点：
- 最系统、最完整

缺点：
- 对当前项目体量偏重，维护成本较高
- 容易写出超前于实现的“理论文档”

---

## 设计决策

采用 **方案 A**，并坚持以下原则：

- **只写仓库已实现事实**：以当前代码、测试、配置、部署文件为准，不为未来功能预留大段描述。
- **避免与 PRD 重复**：业务目标、角色背景、范围界定继续以 `docs/requirements/` 为准；本次侧重实现认知与维护实践。
- **文档即导航**：每份文档都围绕“开发者下一步该看哪、改哪、验哪”组织。
- **优先高频问题**：优先补“入口在哪”“数据如何流动”“环境怎么配”“改动后跑哪些校验”。

## 成功标准

- 文档全部使用中文，且语义准确、可执行。
- 每条关键流程都能从文档映射到实际代码路径。
- 所有命令、环境变量、模型名、路由名均与仓库当前状态一致。
- README、runbook、新增开发文档之间职责清晰，不出现大段重复内容。

---

## 验证方式

- 编写前后对照现有代码、测试与配置文件逐项核对事实。
- 完成后执行 `python manage.py check`，确保文档补充未引入配置层误写或误导。
- 自检所有引用路径、文件名、命令、环境变量与模型字段是否与仓库一致。
